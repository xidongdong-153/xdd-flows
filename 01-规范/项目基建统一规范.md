# 项目基建统一规范

这套规则用于你后续所有项目。
目标很简单：项目结构、文档风格、Agent 读取入口尽量保持一致，这样你自己切项目快，Agent 接项目也快。

## 固定做法

1. 每个项目都在 `04-项目库` 建一张项目卡片。
2. 每个代码仓库都至少补齐 `README.md`、`AGENTS.md`、`docs/architecture.md`、`docs/development.md`。
3. 项目有新增模块、接口、流程变化时，同步更新仓库文档和项目卡片。
4. 项目说明默认写中文，命令、路径、技术名保留常见英文写法。
5. 文案只写当前实现和当前用法，不写历史过程，不保留旧方案描述。

## 项目卡片最少要写什么

每张项目卡片至少补齐这些内容：

- 项目定位
- 仓库路径
- 技术栈
- 目录约定
- 文档入口
- Agent 启动说明
- 当前阶段
- 当前任务

## 统一字段

项目卡片统一维护这些属性，方便 `Bases` 汇总：

- `project_name`：项目名
- `status`：当前状态，比如 `待启动`、`进行中`、`暂停`、`归档`
- `stage`：当前阶段，比如 `规划中`、`开发中`、`联调中`、`收尾中`
- `repo_path`：仓库路径
- `doc_status`：文档状态，比如 `未开始`、`整理中`、`完整`
- `agent_ready`：是否已经整理到能直接给 Agent 读
- `primary_stack`：主技术栈

## 文案要求

- 先写清这份内容用于什么、放在哪里、什么时候改、怎么用。
- 同一套项目内的术语和标题尽量固定，不要一份写“模块说明”，另一份写“功能设计”却指同一件事。
- 如果一句话能说清，就不要堆一串近义词。
- 说明性文案统一按 [[01-规范/说明性文案规范]] 维护。
- 只要项目里出现 README、`docs/`、JSDoc、代码注释、错误提示、提示词或脚本说明，都按这份文案规范来写。

## 输出要求

- 回答和文档说明先给结论，再给修补方案或使用方式。
- 简单问题不做汇报式拆解，不重复背景。
- 多项内容优先整理成 Markdown 表格，尤其是评审、对比和任务分组。
- 输出结尾明确写出这次用到的 skill。

## 维护规则

- 仓库目录调整后，先改项目卡片里的“目录约定”和“文档入口”。
- 开发流程调整后，先改这里和 `02-工作流` 下的流程笔记。
- 某个模板已经稳定可复用，就优先改模板，不要每个项目单独改一套。
