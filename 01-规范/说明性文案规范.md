# 说明性文案规范

这份规范用于统一 README、`docs/`、JSDoc、代码注释、接口说明、错误提示、提示词和脚本说明的写法。

## 先做什么

写说明性文案前，先按下面顺序看上下文：

1. 最相关的 `AGENTS.md`
2. 当前项目的 `README.md`
3. 当前项目的 `docs/`
4. 当前任务直接相关的模块说明

先确认这次要写的是哪一类内容，再开始下笔。

## 固定写法

- 默认使用中文。
- 只写当前实现，不写历史过程、迁移说明、兼容背景和旧方案对比。
- 先写清这份内容用于什么、放在哪里、什么时候使用、怎么调用、输入输出是什么。
- 文案尽量让第一次接手的人也能看懂，不假设对方已经知道上下文。
- 同一任务里的术语、语气、标题和列表格式保持一致。

## 输出规范

- 直接给结论和修补方案，不先复读背景。
- 简单问题不要拆成“证据、分析、结论”这类多段汇报。
- 解释尽量用短小、直接、好懂的中文。
- 不显示 `P0`、`P1` 这类等级。
- 多项内容、评审结果、差异对比和任务清单，优先用 Markdown 表格输出。
- 结束时明确说明这次用到的 skill。

## README 和 `docs/` 怎么写

- 按“目的 -> 位置 -> 规则 -> 示例”组织内容。
- 先说明这个模块或目录做什么，再说明入口文件、目录位置和使用方式。
- 示例只保留当前能直接照着用的命令、路径和代码。
- 写模块关系时，直接写“调用什么”“使用什么”“到哪里改”。

## JSDoc 和代码注释怎么写

- 只说明当前函数、类、方法或代码块的作用。
- 写清参数、返回值、调用时机和必要限制。
- 不在注释里写重构过程、旧实现和兼容原因。

## 错误提示、接口说明、提示词怎么写

- 先说明什么情况下会触发。
- 再说明用户会看到什么、要怎么处理。
- 提示词和脚本说明直接写执行目标、输入内容、输出结果和使用方式。

## 尽量这样表达

- 用于…
- 放在…
- 这里负责…
- 通过…处理…
- 调用…
- 使用…
- 接入…
- 统一在…维护
- 以…为准

## 尽量不要这样写

不要主动写这些内容：

- 历史对比
- 迁移说明
- 兼容背景
- 旧接口行为
- 过渡方案

尽量不要用这些词：

- 原来
- 之前
- 旧版
- 迁移自
- 替换为
- 现在改成

尽量不要用这些抽象说法：

- 真相源
- 类型基线
- 消费边界
- 职责边界
- 能力收敛
- 统一出口
- 对外暴露

如果真的不得不用抽象词，后面马上补一句具体说明：在哪个目录、改哪个文件、由谁调用、输入输出是什么。

## 自查顺序

1. 有没有写到旧实现、历史过程或兼容背景。
2. 有没有先把“做什么、放哪里、怎么用”讲清楚。
3. 第一次接手的人能不能顺着看懂。
4. 有没有为了显得专业而写成抽象黑话。
5. 术语、标题和列表格式是不是统一。
6. 示例、命令、路径和返回结果能不能直接使用。
7. 输出是不是先给结论，最后有没有明确说明用到的 skill。
