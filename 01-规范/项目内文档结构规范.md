# 项目内文档结构规范

这份笔记用于约定每个代码仓库内的基础文档怎么建、各自写什么、什么时候更新。

## 推荐结构

```text
仓库根目录
├── README.md
├── AGENTS.md
└── docs
    ├── architecture.md
    ├── development.md
    ├── modules
    │   └── xxx.md
    └── api
        └── xxx.md
```

## 各文件怎么用

### `README.md`

- 用于写项目简介、启动方式、常用命令、目录入口。
- 第一次接手项目的人，先看这里。
- 这里放最常用、最需要快速找到的内容。

### `AGENTS.md`

- 用于写 Agent 在当前仓库里的执行规则。
- 这里放读取顺序、硬性约束、技能调用要求、文案要求和必须遵守的流程。
- 只保留当前做法，不写过去怎么做。

### `docs/architecture.md`

- 用于写项目结构、目录职责、模块关系。
- 这里重点写“去哪里看、改哪里、模块之间怎么接”。
- 结构变化时优先更新这份文档。

### `docs/development.md`

- 用于写开发流程、命令、环境变量、调试方式、构建和测试入口。
- 这里的命令要能直接照着运行。

### `docs/modules/*.md`

- 用于写某个模块的入口、目录、调用方式、返回结果和修改位置。
- 模块变复杂后再拆，不需要一开始就铺很多空文件。

### `docs/api/*.md`

- 用于写接口分组、请求响应、鉴权要求、错误处理和联调说明。
- 对外接口、多人联调接口优先补。

## 更新时机

遇到下面这些情况，就同步改文档：

- 目录结构变了
- 启动命令变了
- 环境变量变了
- 模块入口变了
- 接口行为变了
- Agent 读取顺序变了

## 写法要求

- 文档里尽量写路径、文件名、命令和返回结果。
- 面向自己和 Agent 时，都按“第一次看也能快速找到位置”来写。
- 说明性文案统一按当前实现写，不补迁移过程和旧方案对比。
- README、`docs/`、接口说明、错误提示和提示词，统一按 [[01-规范/说明性文案规范]] 的写法整理。
